## 1. プロダクト概要

### 1.1 プロダクトビジョン
ソリューション営業企業の**商機転化率を 30〜60% 向上**させる、AI 駆動の案件インテリジェンス基盤。
単なる CRM／SFA ではなく、「受信した案件情報を即座に理解・分析・準備・実行」まで一気通貫で自動化するプラットフォーム。

> **前提条件**: 自社のソリューション・知識・過去事例・製品資料が豊富な企業ほど、AI との相乗効果が最大化する。

### 1.2 ターゲットユーザー
| ユーザー層 | 特徴 |
|---|---|
| SIer・ITコンサル | 多様な要件書・提案書・事例を保有 |
| エンジニアリングサービス企業 | 技術的ナレッジが豊富・案件量が多い |
| BPO・業務改革コンサル | 業種別ワークフロー知識が深い |
| 専門商社・ソリューション代理店 | 製品知識・顧客業種ナレッジが資産 |

### 1.3 技術スタック（V10.5.1）
| 項目 | 内容 |
|---|---|
| バックエンド | Google Apps Script (GAS) V8 Runtime |
| AI エンジン | DeepSeek-V3-2（Volces Ark API）|
| フロントエンド | Vanilla JS + Tailwind CSS（GAS Webapp として配信） |
| データストア | Google Spreadsheet（Case_infor / Review_Buffer / Meeting_Records 等） |
| 外部連携 | Yeeflow CRM API / Gmail API / 任意 CRM（REST API） |
| 認証 | Google Apps Script ドメイン制限（ALLOWED_DOMAIN） |
| シークレット管理 | Script Properties + sync-secrets-to-gas.sh |

## 5. 商機転化率 30〜60% 向上の根拠

### 5.1 転化率向上のメカニズム（4段階）

**前提**: 自社のソリューション知識・過去事例・製品資料が豊富な企業に適用した場合。

```
現状の転化率ロス原因
①　案件理解の遅延・誤読（担当者差）     → LMS: AI が数分で統一品質に構造化
②　重要案件の見落とし・後手対応         → LMS: 着信から数分で全件スコアリング通知
③　商談前準備の属人的な品質ムラ        → LMS: Agenda / Hearing / Demo を自動生成
④　Knowhow の属人化・再現できない      → LMS: 判断ログ・事例・提案書を AI が照合
```

### 5.2 定量的な根拠ロジック

| 改善要因 | 効果 | 転化率への寄与（推定） |
|---|---|---|
| 初期分析の均一化 | 見落とし・誤判断ゼロ | +5〜10% |
| 初回レスポンス速度向上 | 競合より早い接触 = 信頼形成 | +5〜15% |
| 商談前準備の品質均一化 | 初回商談の成功率向上 | +10〜20% |
| Knowhow の再現・照合 | 類似案件の勝ちパターン再現 | +10〜15% |
| **合計（相乗効果あり）** | | **+30〜60%** |

> 注: 転化率の「ベースライン」が低い企業ほど向上幅が大きい。  
> 既に高い組織（転化率 50%以上）では +10〜20% 程度の改善が現実的。

### 6.3 LMS の唯一性（競合が持たない差別化ポイント）

#### ① 「案件理解の深さ」が競合と根本的に異なる
競合製品が提供する AI は主に「案件管理の自動化・入力補助」。
LMS の AI は「**顧客のビジネス文脈を読み解く需要分析エージェント**」—— 意思決定構造・担当者影響力・業界ポジション・タイムライン・公開情報の意図まで多角的に分析する。

> 例: Mazrica の AI は「次のアクション示唆」。LMS の AI は「この担当者の役職から最終決裁者は別にいる可能性が高く、経営課題への訴求が必要」まで推論する。

#### ② 「商談前ドキュメントの丸ごと自動生成」は市場にない
Agenda / Hearing Sheet / Workflow 分析 / Demo シナリオ / 提案書骨格の**一気通貫自動生成**は日本市場の競合製品にない機能。

#### ③ 「自社 Knowhow の資産化＋AI 自動活用」が本質的差別化
競合は汎用的なデータベースを使う。LMS は自社の提案書・成功事例・製品資料・判断ログを AI に学習させ、**「自社のベテランの判断を再現する」**仕組みを実現する。

#### ④ プロンプトのノーコード管理で継続的な精度向上
Prompt_Config でプロンプトを担当チームが自由にチューニング。エンジニア不要で分析精度を業種・商材に最適化できる。

#### ⑤ 既存 CRM を置き換えない「前段インテリジェンス」設計
CRM の置き換えを求めず、**既存 CRM の「前段処理」**として位置付け。CRM のデータ品質を向上させつつ、追加インフラも不要。

## 9. 既知の制約・注意事項

1. **GAS の実行時間制限**: 1回の実行は最大 6分。大量処理はバッチ分割が必要
2. **GAS の権限承認**: 長期間未実行時に `Authorization required` エラーが発生 → GAS エディタで手動実行して再承認
3. **絵文字**: GAS HTML で UTF-8 絵文字が文字化け → 必ず HTML エンティティ使用
4. **AI レスポンス時間**: DeepSeek API は数秒〜10秒。生成中はボタン disabled が必須
5. **データ量**: スプレッドシートは数千行規模が限界 → 大規模化時は DB 移行を検討
6. **公開情報取得**: 現在は受信文書内のテキストのみ解析。外部サイトのクロールは未実装（法的制約も考慮が必要）
